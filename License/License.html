Mit License
 Permissive (Non-Copyleft). Simple and short, grants maximum freedom to use, copy, modify, and distribute.
 Must include the original copyright notice and a copy of the license text in all distributions.
 Yes
 Yes (Safe for proprietary code integration)
 Safest. Requires only simple attribution/notice. No copyleft effect.
BSD Licenses (e.g., 2-Clause/3-Clause)
 Permissive (Non-Copyleft). Very similar to MIT, originating from UC Berkeley.
 Must retain the copyright notice and disclaimer of warranty. 3-Clause adds: cannot use contributor's name to endorse the derived produc
 Yes
 Yes (Safe for proprietary code integration)
 Safest. Requires only simple attribution/notice. No copyleft effect.
Apache License 2.0 (Apache)
 Permissive (Non-Copyleft). More detailed, provides explicit patent grant and protection against patent claims.
 Must include the license, copyright notice, and NOTICE file (if present). You must state all significant changes to the files.
 Yes
 Yes (Highly favored for corporate closed-source projects)
 Safe. Requires attribution, notice file, and stating changes. Explicit patent grant is a benefit for commercial use. No copyleft effect.
Academic Free License (AFL)
 Permissive (Non-Copyleft). Legally robust, explicitly covers patents.
 Must provide the Source Code of the AFL-licensed component to recipients upon distribution. Must retain all notices.
 Yes
 Yes (But you must provide source of the AFL-licensed part)
 Conditional. Non-copyleft, but requires you to provide the source code of the component itself upon distribution, which adds distribution complexity.
GNU General Public License (GPL) (Strong Copyleft)
 Strong Copyleft ("Viral"). Guarantees software freedom; focuses on ensuring the work and all derivatives remain open.
 Strong Copyleft ("Viral"). Guarantees software freedom; focuses on ensuring the work and all derivatives remain open.
 Yes
 NO (This is the strong Copyleft effect)
 Incompatible. Strong copyleft means your entire derivative work (your project) must be released under the GPL.
GNU Lesser General Public License (LGPL Weak Copyleft. Designed for libraries; allows proprietary software to link to it (e.g., dynamically linked LLM library). If you modify the LGPL component itself, those modifications must be released under LGPL. You must allow users to re-link your proprietary application with a modified version of the library.
 Yes
 Yes (As long as you use Dynamic Linking and provide re-linking instructions)
 Conditional. Weak copyleft. Requires dynamic linking to the library and must allow users to substitute the library. Compliance requires technical care.
Mozilla Public License (MPL) (e.g., v2.0)
 Weak Copyleft (File-based). More permissive than LGPL.
 Any modified MPL-licensed file must remain under the MPL. However, code in other, separate files can be proprietary.
 Yes
 Yes (Source code disclosure is limited to the MPL files only)
 Conditional. Weak (file-based) copyleft. Requires only modifications to the MPL files to be open, but requires careful file separation.
Artistic License (e.g., v2.0)
 Permissive/Weak Copyleft (depending on version). Often used for Perl. Allows proprietary derivatives but with an obligation to make modifications available if redistributed.
 Must keep the Original Author's identity intact. Modifications must be made publicly available if the work is redistributed.
 Yes
 Yes (Generally safe, but has specific rules for distributing modifications)
 Conditional. Compliance rules vary and can be ambiguous, often requiring modifications to be made available.

Creative Commons (CC BY) Primarily for content (datasets, documentation, images), not generally recommended for software code/LLMs. BY = Attribution.
 Must provide Attribution (credit) to the original creator.
 Yes
 Yes (No copyleft, but focus is on content, not code)
 Conditional. For content/data. Safe if applied only to content/data and not the underlying code/engine, and requires attribution.
Creative Commons (CC BY-NC)
 NC = Non-Commercial. Specifically prohibits use for a commercial purpose
 Must provide Attribution. Prohibits use where the primary purpose is commercial advantage or monetary compensation.
 No
 N/A (Cannot be used commercially)
 Incompatible. Explicitly prohibits commercial use, which is the goal of your closed-source project.
Classic Proprietary (EULA)
 The license for traditional, non-open-source software (e.g., Microsoft Word, commercial SaaS). EULA = End-User License Agreement.
 Users have only the rights explicitly granted (usually just to run the program). Modification, reverse engineering, and redistribution are strictly prohibited.
 N/A (The product itself is proprietary) Yes (Your proprietary code is safe)
 Incompatible. If this license is on your product, it's good. If it's on a component you want to use, you likely cannot use it as it strictly prohibits modification/integration.
Sun Community Source License (SCSL)
 Source-Available, NOT Open Source. Created by Sun Microsystems; allows viewing source code under restrictive conditions.
 Strictly limits redistribution and modification. Often requires separate fees and approval for commercial distribution.
 Highly Restricted (Requires commercial agreement)
 Yes (But you're subject to Sun's/Oracle's restrictions)
 Incompatible. Highly restrictive, proprietary terms, and requires separate commercial agreements/fees for distribution. Not an open-source license.
Microsoft Shared Source Initiative
 Source-Available, NOT Open Source. A framework for Microsoft to share source code under varying non-OSI-approved terms.
 The most restrictive licenses in this initiative (like MS-RSL) allow viewing for reference only (e.g., debugging) and strictly prohibit modification or redistribution.
 NO (Highly restricted use)
 Yes (But you cannot integrate or distribute the code)
 Incompatible. Highly restrictive; often only allows code viewing for reference/debugging and strictly prohibits modification or redistribution in your product. Not an open-source license.
