LANGSERVE ‚Äì INTRODUCTION

Langserve
Langserve is used to deploy langgraph agents. before deployment first some hurdles need to be passed and thought about being:

* resource management llms are computationall intensive, high memory and processing power is required to run a llm

* latency and performance. LLMs can be slow to process requests, especially under heavy loads. Optimizing performance involves strategies such as model compression, efficient serving frameworks, and possibly offloading some processing tasks to edge devices.

*Once deployed, continuous monitoring of the LLM application is necessary. This includes tracking performance metrics, detecting anomalies, and managing model drift

*Integrating LLMs with existing systems and workflows can be challenging. Ensuring compatibility with various software environments, APIs, and data formats requires meticulous planning and execution.

*The high computational demands of LLMs can lead to significant operational costs. Balancing performance with cost efficiency is a vital consideration. Strategies to manage costs include optimizing resource allocation, using cost-effective cloud services, and regularly reviewing usage patterns to identify areas for savings
There are several approaches to deploying LLMs into production, each with its advantages and challenges.:

* On-premises deployment involves hosting the LLM on local servers or data centers, offering greater control over data and infrastructure but requiring significant investment in hardware and maintenance.

* Cloud-based deployment leverages cloud services to host the LLM, providing scalability, flexibility, and reduced upfront costs, although it can introduce concerns about data privacy and ongoing operational costs.

* Hybrid deployment combines on-premises and cloud resources, offering a balance between control and scalability, allowing organizations to optimize performance and cost based on their specific needs.

Langserve VS FastAPI
While LangServe is a specialized tool designed to simplify the process of deploying LLM applications making it quicker to set up with less lines of code than using e.g. FastAPI,.

Due to the company behind langserve wanting to make a profit langserve is no longer in development (replaced with a paid cloud service). Therefore in the future the deployment of complicated agents and graph might have to directly be done using FastApi without langserve due to competability issues.

Another reason to use FastApi directly instead of Langserve is because deployment using langserve will create a set amount of default API endpoints, which using FastApi is fully customizable.

Deploying using Langserve

Installation
you can install langserve using pip through "pip install "langserve[all]" if you just want the client or server replace "all" with "client" or "server"

Setup
using the LangChain Commandline interface  create a new application using the following command and add the third-party packages (using Poetry) and set up environment variables

Example
app new <app_name> #this creates a new directory

#go to the directory
<app_name>

#add third party packages
add langchain-openai langchain langchain community

#set environment variables
OPENAI_API_KEY="<api key>"

Setup Server
The "server.py" will contain the main logic for the LangServe application. it utilises FastApi to create the webserver and uvicorn to host the app. langserve itself is used in the way of its add_route() classes which are used to create "routes" to the fast api app (thereby creating fastapi endponts)

the add_routes() requires 3 arguments
1- the fastapi app
2 the langgraph  graph_builder.compile() compiled graph, or straight up a langchain supported model or other langchain runnables
3- path = the endpoint path.

you can add as many routes (endpoints) as needed

in the end Uvicorn is used to host the Fastapi app and can thus be accessed using Curl (see Uvicorn notes) (as it is in the _name_ == __main__ part this can be used for testing purposes or to run fully through python.

Example
fastapi import FastAPI
from langchain.prompts import ChatPromptTemplate
from langchain.chat_models import ChatOpenAI
from langserve import add_routes
import uvicorn

app = FastAPI(
    title="LangChain Server",
    version="1.0",
    description="A simple API server using Langchain's Runnable interfaces",
)
# Define a route for the OpenAI chat model
add_routes(
    app,
    ChatOpenAI(),
    path="/openai",
)
# Define a route with a custom prompt
summarize_prompt = ChatPromptTemplate.from_template("Summarize the following text: {text}")
add_routes(
    app,
    summarize_prompt | ChatOpenAI(),
    path="/summarize",
)
if __name__ == "__main__":
    uvicorn.run(app, host="localhost", port=8000)

Running the server (using Poetry)
Going back to the Langchain CLI the application can also be run using Poetry with the command " run langchain serve --port=<port number>"
ths will run the langserve app server on the given port
once application startup is complete you can go too "http://<server ip>:<server port>/summarize/playground/"  to execute runables, view streaming output and observe intermediat steps straight from your browser.
the playground  facilitates easy interaction for debugigng purposes.

as the underlying app server is still FastApi you can go to "http://<server ip>:<server port>/docs"  to see the fastapi generated documentation. allowing for aditional testing of endpoints and reading the documentation.

Logging
logging is setup almost idenitlaly as for fastapi bu rather than using the Uvicorn logger it uses the "lanchain_server" logger. other than that it uses pythons logging library. the log levels are in order (debug, info, warning,error, critical) and by using logger.<log_level> you can write logs throughout the code.

Example
import logging

# Configure logging
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("langchain_server")

app = FastAPI()

# Example of logging within a route
@app.get("/status")
async def status():
    logger.info("Status endpoint was called")
    return {"status": "Running"}
