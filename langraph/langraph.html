LangGraph ‚Äì INTRODUCTION

LangGraph
LangGraph is an branch of the LangChain ecosystem. specifically for designing agentic systems, or systems including agents.
LangGraph can enhance tool use even further by structuring tasks in workflows called graphs. In these graphs, tasks called "nodes" are connected by rules called "edges". For example, a database query node can link to a document retrieval node, with an edge pointing to which document is retrieved.

Agent
Agents use llms to take actions.AI agents exhibit a degree of autonomy, memory. making it a software system that uses artificial intelligence to autonomously pursue goals and complete complex, multistep tasks on behalf of a user or system.

Tools
Agents often use tools.Tools are functions called by the agent (like run code, load document, call upon an api ect).

ReAct agent
ReAct stand for reasoning and acting. a ReAct agent prompts the model using a repeated loop of thinking, acting, and observing. e.g. asking the weather to an agent it first interpertrs the questions than use its tool to call for the weather api, than observers the results and then gives the awnser.

besides making your own agents provide langgraph a variaty of prebuilt agents among which a ReAct agent in the form of the create_react_agent() function.
the create_react_agents() requires variables 1 containing the llm and the tools to be used

The langchain community allready made toolkits for a variety of pruposes among which doing math, these can be imported using the load_tools() function which takes 2 variables a list of tools and the llm to use these tools.

just like chains agents can be executed using the .invoke() function. the invoke methord returns a dictonary containing the key value pair messages and a list of messages that are retruned among wihich the input (humanMessage), the ais translation of the message to the tool (AiMessage), the tools response (ToolMessage) and the final output message of the ai (AiMessage) to

Custom tools
Tools in LangChain must be formatted in a specific way to be compatible with agents. They must have:
* a name, accessible via the .name attribute. ( by default this is the function name)
* a description, which is used by the LLM to determine when to call the tool. For example In case the of a math tool, if the LLM interprets the task as being a math problem, it will likely call this tool based on its description. (by default this is the docstring)
* return_direct paramater which indicates whether the agent should stop after invoking this too (set to default by default)

you can also print the argument names which shows the names an expected datatypes using the .args attribute

To turn a python function into a tool the @tool decorator is used

Building graphs for chatbots
In LangGraph, the graph state organizes the agent's order of tasks, such as tool usage and LLM calls, into a workflow. The graph uses an agent state to track the agent's progress as text to help determine when a task is complete.

The build a chatbot that awnsers questions. We only need three nodes and two edges to get started. In LangGraph, nodes represent functions, like generating a response or calling a tool, while edges define the connections between these nodes.
We need a start node, chatbot node and end node.
The start and end node can be directly imported from langraph.

To get started,
we'll set OpenAI as our llm.
Then, we'll create an agent state called State that accepts TypedDict to structure messages.
We'll define messages as a list that will store all text interactions with the chatbot. The Annotated type, combined with add_messages,  ensures that new messages are added to this list with metadata.
Next, we'll initialize a StateGraph() called graph_builder and pass in our State structure.
The graph builder will organize the nodes and edges in the chatbot's workflow.

Adding memory to a LangGraph chatbot
to add Memory to a langGraph chatbot, 
1. First, we'll import MemorySaver from langgraph.checkpoint.memory to handle memory storage within our graph.
2. Then, we'll create a MemorySaver instance called memory, which will act as our checkpoint for storing messages.
 3. Next, we'll compile the graph with graph_builder.compile(), passing memory as the checkpoint, so the chatbot retains conversation context.
4. use the newly compiled graph in a stream function

Example (single session memory)
from langgraph.checkpoint.memory import MemorySaver

Create stopping function

stopping function decides whether to call tools or end the conversation if there is no tool call in the last message.
in its simpelest form the function uses the MessageState to define the state of the stopping function and retrieve if the last message contained any calls if not returns the END function to end the graph.

Example
from langgraph.graph import MessageState, START, END

def should_continue(state: MessageState):
     last_message = state["messages"][-1]
     if last_message.tool_calls:
          return "tools"
     return END

Create dynamic tool caller function
A dynamic tool caller that returns a tool response if a tool call is present,or invokes the LLM with the chatbot node if there are no tool calls.
in simple terms it check if the last call was an ai message tool call if so returns the response if not invokes the llm to continue.

Example
def call_model(state: MessageState):
     last_message = state["messages"][-1]
     if instance(last_message, AIMessage) and last_message.tools_calls:
           return {"messages": [AIMessage(content=last_message.tool_calls[0]]["response"])]}
     return {"messages": [model_with_tools.invoke(state["messages"])]}

Use custom functions to create a graph
1 the graph state is created using the MessagesState being passed to StateGraph()
2 add the created nodes to the workflow (bots and tools) with the dynamic tool caller function as the "chatbot" node
3 connect the START node to the chatbot using add_edge()
4 use the stoppng function as the conditional_edge by providing the "chatbot" the should_continue function and list of options
5 make the tools node loop back to the chatbot toe refine the response (so it reads better)
6 setup memory that the functions can call upon to check the last call
7 compile the work flow
8 use the graph in a def (using stream_mode = messagen and setting flush = true for live returning of responses)

Python running tool
While still experimental langchain has the ability to run python code using the PythonREPL
this is done through
1. first setting the repl
2. and then while using the .run() method on it providing the query

Example
from langchain_core.tools import tool
from typing import Annotated
from langchain_experimental.utilities import PythonREPL

repl = PythonREPL()

@tool
def python_repl_tool(
    code: Annotated[str, "The python code to execute to generate your chart."],
):
    """Use this to execute python code. If you want to see the output of a value,
    you should print it out with `print(...)`. This is visible to the user. The chart should be displayed using `plt.show()`."""
    try:
        result = repl.run(code)
    except BaseException as e:
        return f"Failed to execute. Error: {repr(e)}"
    return f"Successfully executed the Python REPL tool.\n\nPython code executed:\n\`\`\`python\n{code}\n\`\`\`\n\nCode output:\n\`\`\`\n{result}\`\`\`"

code = f"""
import numpy as np

arr = np.arange(0, 9)
print(arr)
print(2 * arr)
"""   

print(python_repl_tool.invoke({"code": code}))
