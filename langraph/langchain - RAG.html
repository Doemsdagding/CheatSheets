LANGCHAIN – RAG

Retrieval Augmented Generation (RAG)
Pre-trained language models don't have access to external data sources - their understanding comes purely from their training data.
In RAG, a user query is embedded and used to retrieve the most relevant documents from the database. Then, these documents are added to the model's prompt so that the model has extra context to inform its response.

There are three primary steps to RAG development in LangChain.
1: is loading the documents into LangChain with document loaders.
2: is splitting the documents into chunks that we can index and process individually.
3: is encoding and storing the chunks for retrieval, which could utilize a vector database if that meets the needs of the use case.

Rag 2: Document splitting
Document splitting splits the loaded document into smaller parts, which are also called chunks. Chunking is particularly useful for breaking up long documents so that they fit within an LLM's context window.
One naive splitting option would be to separate the document by-line. but as sentences are often split over multiples context might be lost. for this reason chunk overlap is often implemented.resulting in that part of the setence is in both chuinks. the size of the overlap can be tweeked to improve understanding.
There are many splitting methods examples are CharacterTextSplitter and ResursiveCharacterTextSplitter

Example
quote = """One machine can do the work of fifty ordinary humans. \nNo machine can do the work of one extraordinary human."""
len(quote) #103
CharacterTextSplitter is a method that splits on a sperator, e.g. a period, first then if chcunks are smaller than the given chunk size it uses the chunks_size and chunk_overlap to combine the small chunks into larger once. WARNING if chunks are larger than the chunk_size it does not split the chunk up.

for this the CharacterTextSplitter class is used which requires:
seperater = the intended seperator for initial splitting
chunk_size = amount of chracters per chunk
chunk_overlap = the amount of overlap chracter

after this the splitter can be used, using the .split_text() function which takes the text to be split.

Example (CharacterTextSplitter())
from langchain_text_splitters import CharacterTextSplitter

ct_splitter = CharacterTextSplitter(
     seperator=".",
    chunk_size =  24
    chunk_overlap = 3 )

docs = ct_splitter.split_text(quote)
To resolve the problem of chunks remaining to large the RecursiveCharacterTextSplitter class can be used instead. its basicaully the same as the characterTextSpliter but instead of 1 seperater it takes a list of seperators. and it goes down that list 1 one by 1 till the chunks are small enough to fit the chunk_size.

Example ( RecursiveCharacterTextSplitter())
from langchain_text_splitters import RecursiveCharacterTextSpliter

rc_splitter = RecursiveCharacterTextSplitter(
    seperator = ["n\n\n", |\n", " ", ""]],
    chunk_size = 24,
    chunk_overlap = 3)
docs = rc_splitter.split_text(quote)
To split documents instead of text the .split_ducments() can be used. which works teh same as the split_text() function.

RAG 3: Storage
The most optimal way to store the documents is to store in a vector database e.g. Chromadb (see chromadb notes).  This requires embedding our text documents to create vectors that capture the semantic meaning of the text.
There are many vector databases and Embedding models e.g. chromadb and openaiembeddings( see openai advance notes) available directly in LangChain. which can be used in the chains.

these generally work the same as their own libraries but function/class might be called different.

for chromadb there is an additional funtion classes which lets you import the chunked docs directly into the vectordatabase. being the from_documents() class which combined the client and create_collection functions of chromadb. from_documents requires 2 variables
documents = the document objects
embedding = the embedding function
furthermore it has several optional variables among which but not limited too:
presist_directory =  takes a file path in the form of a string to store the db at.without it the db is saved in local storage
client =a chromadb.client can be given if the client has been premae using the chromadb pythonpackage (see chromadb notes)
collection_name = the name of the collection default to langchain.
ids = list of uniue string ids for the documents (must be same amount as number of documents.
collection__metadata = a dict same as metadata in chromadb package.

Example

RAG 4: Retrieval
A user query can be embedded to retrieve the most similar documents from the database and insert them into the model prompt.

to integrate the database with other langchain components the vectordatabase needs to e converted into a retriever using the .as_retriever() function
the .as_retriever() function requires 2 variables
search_type = the search method default is "similarity" which ranks documents purely on how ismilar their vector embbedigns are to the query vector, other options are "mmr" maximal marginal relevanse which also takes into account that the result should be diverse so first if its relevant tkaes the best of that and the rest should be relevant but not to similar to the first one.
search_kwargs = adictionary containing depending on the search_type different kwargs need to be iven for similarity only a "k" key with a value of the amount of results is needed for mmr also "fetch_k" = the initial amount of results before selecting just the best and diverse of those and "lambda_mult" the balance of diversity and relevance in teh form of a number 0 to 1 e.g. 0.5

Example
.... (see rag 3)

retrieve = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 10, "fetch_k": 20, "lambda_mult": 0.5}
)

Document loaders 4: SQLDatabaseLoader
To load data from a sql database  instead of a vector database the SQLdatabase and SQLDatabaseLoader can be used. the SQLDatabase class takes a database engine from the sqlalchmy library.

the SQLDatabaseLoader takes a query and an SQLDatabase class the query is than run and the output can be loaded using the regular .load() function.

Example

from sqlalchemy import create_engine
from langchain_community.utilities import SQLDatabase
from langchain_community.document_loaders import SQLDatabaseLoader

# Minimal database setup (required for a runnable example)
engine = create_engine("sqlite:///:memory:")
conn = sqlite3.connect(engine.url.database)
conn.executescript("CREATE TABLE items (name TEXT); INSERT INTO items (name) VALUES ('Widget A');")
conn.close()

# The core LangChain steps

query = "SELECT name FROM items"
loader = SQLDatabaseLoader(query=query, db=db)
documents = loader.load()

print(documents[0].page_content)

Retrieval 2: Sparse retrieval
while de basic retriever (see rag: retrieval)  uses a dense retrieval method (encoding chunks as a single vector) which can be used to capture semantic meaning but is more ocmputionally expesnive.  using a sparse retreival method (encode usign word mathing ) being more precise explainable and better at rare-word handeling.but has the con of generalizability.

to use a sparse vector method the BM25 encoding method can be used. it measures a specific terms frequency in a document and rarity in other documents. to identiy if this document is what is searched for.

the BM25Retrives accepts 2 arguments
chunks = containing the data to retreive from and
k = which is the number of item returned by the retriever when invoked.
* instead of using from_texts also from_documents can be used  which than requires teh chunks to be documents.

Example
from langchain_community.retrievers import BM25Retriever

chunks = ["python was created by guido van rossum and released in 1991", "python is a popular language for machien elarning (ML).", "The PyTorch library is a popular Python library for Ai and Ml." ]

bm25_retriever = BM25Retriever.from_texts(chunks, k=3)

results = bm25_retriever.invoke("when was python created?")
print("Most Relavant Document: \n" + results[0].page_content)

RAG evaluation 2: RAGAS
RAGAS was designed to evaluate both the retrieval and generation components of a RAG application.(faithfulness, anwserrelavnacy, ccontext precision and cotnext recall.
Faithfulness assesses whether the generated output represents the retrieved documents well. It is calculated using LLMs to assess the ratio of faithful claims that can be derived from the context to the total number of claims. Because faithfulness is a proportion, it is normalized to between zero and one, where a higher score indicates greater faithfulness.

to evalue the RAG using RAGAS the ragas library can be used using the EvaluatorChain wit the metric you want to evaluate the rag can be evaluate for each of the 4 metrics.

Example
from langchain_openai import ChatOpenAi, OpenAIEmbeddings
from ragas.integrations.langchain import EvaluatorChain
from ragas.metrics import faithfulness

llm = chatOpenAi(model = "gpt-40-mini", api_key = "****")
embeddings = OpenAiEmbeddings(model= "text-embedding-3-small", api_key="****")

faithfulness_chain = EvaluatorChain( metric= faithfulness,
    llm=llm,
    embeddings=embeddings
)

eval_result = faithfulness_chain( {
"question" : "How does the rag model imrpvoe question answering with llms?",
"answer": "The rag model imrpoves question anwserign by comining the retrieval of ducments... ",
"ccontexts": [ "the rag model integrates document retreival with llms by first retreiving relevant pasages..", "By incoperating retrieval mecchanisms rag leverage extnal knwoeldge soruces alloweing the ... , ]]})

Graph database
Vector databases have their limitations being that they struggle to capture themes and relationships between document entities. when database grows retrievial can become less efficient, lastly certain document types can be harder to embed into a vector database.

To solve this a graph database can be used. Graphs are great at representing and storing diverse and interconnected information in a structured manner. Entities, like people, places, and sports teams are represented by nodes, and relationships between entities and differentiated based on entity types (e.g. persons and locations).

an example of a graph database is Neo4j which has langchain support builtin the langchain library. to do this the LLMGraphTransformer function used which takes an llm in order to be able create the relations (setting temeprature to 0 is best for consistency.

Afterwards the documents can be converted into the nodes that can be loaded into the database using convert.to_graph_documents() method taking the documents as the argument

Example
#loading and chunking example wikipedia data
from langchain_commmunity.document_loaders import WikipediaLoader
from langchain_text_splitters import TokenTextSplitter

raw_documents = WikipediaLoader(query="Large Language model").load()

documents= text_splitter.split_documents(raw_documents[:3})
# from text to graphs
from langcchain_openai import ChatOpenAi
from langchain_experimental.graph_transformers import LLMGraphTransformer

llm = ChatOpenAi(api_key="******", temperature=0, model_name="gpt-4o-mini")
llm_transformer = LLMGraphTransformer(llm=llm)

graph_documents == llm_transformer.convert_to_graph_document(documents)

Creating a Graph RAG chain
Inorder for the LLM te know what Cypher query to generate the llm need access to the graph schema. for this the GraphCypherQAChain class is used. The user submits a natural language input, it gets translated into a Cypher query, and the generated Cypher is used to query the Neo4j database. Finally, the results obtained from the Neo4j database query are converted back into natural language for the user.

the graphCypherQAChain in cocmbination with the from_llm function takes
llm = the used llm for the graph database
graph = the graph database it self
verbose= bool if set to true the cyper query and result can be viewed.
optionally to improve the graph chain additional arguments can be used
exclude_types = list of node types that hsould be excluded from the result
validate_cypher = bool whicch when set to true makes the chain perform a check on the direction of the relation.

Example
from langchain_community.chains.graph_qa.cypher import GraphCypherQAChain

chain = GraphCypherQAChain.from_llm(
    llm_ChatOpenAi(api_key= "****", temperature =0 ), graph= graph, exclude_types= ["concept'], verbose = True, validate_cypher= True )

result = chain.invoke({"query": "what is the most accurate model?})
print(f"Final answer: {result["result"]}")
