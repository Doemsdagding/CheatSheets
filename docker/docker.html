Virtualization
Virtualization is the process of creating a virtual version of a computing resource.
A virtual machine is called full virtualization.Other levels of virtualization include virtualizing a specific computer component, such as the operating system or storage, which does not result in a virtual machine. often done through vmware.
with environment everything surrounding an it application is meant including software, hardware and the operating system
containers also know as OS-level virtualization isolates user spaces including application dependingsies and application settings, it virtalizes the OS but not the os kernel and hardware. resulting in that a single host can run multiple applications without requiring a dedicated OS for each application.

for building and running containers docker is the common choice for automatically managign and coordianting multiple containers kubernetes is often used.

Orchestration
Container orchestration comes in when you have multiple containers. orchesjtration is the automated management and coordination of multiple containers. for this an orchestrator is used. to achieve this a config file is written which describes the desired autocome. with this config file the orchestrator will allocate cpu memory ect over the containers that need it

To achieve this you can use docker swarm mode or kubernetes.

Starting a container
to start a container from an image you can use the "run" command. by adding the flag:
-it = will run image but create an interactive shell inside the container to communicate with.
-d = will run the container in background so you continue using the terminal (otherwise you have to use "exit" to leave the shell)
--name <container name> = lets you set the container name

Example
docker run <image name>

docker run hello-world

#running ubuntu
docker run -it ubuntu

-a alows you to get additional information about exisiting containers like tthe ports that the container uses.

Example

Creating a docker image 1: basic

Docker file
A docker file is a text file with instruction to create an image.
an image is the blue print to create a container.

a docker file is always execute from top to bottom and if based on another image starts with the "from <dockerimage:(optional version)>" instructions to import this other images.

after which you can continue upon it by customizing it using the "RUN <command>" command

Example
From ubuntu
RUN apt-get update
RUN apt-get install -y python3

Creating the docker image
To create an image you need to build a docker file using the "docker build <file location>" command this builds the image
using the -t flag you can give the image an unique name

Example
docker build -t <name image:v<versionnumbe>r> /location/to/dockerfile

Creating a docker image 2: additional instructions
Besides the "from" and "run" instructions you can use several other instructions in the docker file

adding files (COPY + download)
"COPY <file location input> <file location docker image>" is used to move files from the local machine into the image. this can also be used to copy entire folder to the image by not specifiying a specific file. 

as not everyone might have the file on their local machine there fore they need to download it to do this multiple "run commands" starting with a curl to get the file and then unzip to unzip it and than rm to delete the zip file again. its is best practise to do this in one "RUN" command chained using "\ &&" as otherwise the space the zip file takes doesnt get released when removed

Example
#local file
COPY /project/pipeline_v3/pipeline.py /app/pipeline.py

#downloading files
RUN curl <file-url> -o <destination in image> \ &&
 unzip <dest-folder>/<filename>.zip \ &&
 rm <copy_directory>/<filename>.zip

Shell commands (CMD)
You can also run any shell commands using "CMD <shell-command>" instruction this will execute the shell command on start up of the image. Therefore only one cmd command can exist in a docker file as only the last one will be executed.
But its use is often essential to execute actual script or startup an application which accepts outside connections like a database.

another use case is using it to execute a .sh or .py script that then in turn start multiple applications.

the docker run command can also be overwritten durring the run command of the docker iamge "docker run <image> <shell-command>"

Example
CMD python3 my_pipeline.py
CMD postgres

CMD start.sh
CMD python3 start_pipeline.py
# over writing the CMD command
docker run -it ubuntu bash

Docker caching
When building an image docker will execute the docker image docker file commands. it will keep in its cache all the downloaded materials if after a stop it gets build again it does not check if those files have been updated and thus will not install the latest version.

therefore its best practise to write the dockerfile instructions from least changign to most changing so more cached lines can be used when rebuilding the image

Creating a docker image 3: affect other

 instructions

Most instructions dont have any affect on the instructions that follow them. WORKDIR and USER are exceptions to this.

WORKDIR
THe "WORKDIR" instruction changes the working directory for all following instructions. this can be used to shorts commands like the "copy" and "run" instructions

Example
WORKDIR /home/my_user/with_a_long_name/work?projects/

COPY /projects/pipelien_v3 app/

USER
the "USER" instruction changes the user for all following instructions. this allow for commands that can only be ran by specific users cause they have the rights to access certain folders. Thereby making it possible to not have to run the docker file as root. (by default ubuntu is set to the root user when launched)

Example
From ubuntu
USER repl
RUN apt-get update

Creating a docker image 4: Variables
using the "ARG <var_name> = <Var_value>" instructions a variable can be set that could be used further down in the dockerfile. To than use it in the script the "$<variable_name>" dollar sign can be used.

the big downside of arg is that after the container has been build the variable is no long available to use.

general use of arg is to make it possible to use the same value multiple times without typing it out fully inside the dockerfile.

an additional use case is of varaibles is like mentioned before in the build command to set it there in this case the arg gets overwritten by the build command

Example
ARG path=/home/repl

Copy /local/pah $path
docker build --build-arg project_folder=/rep/pipeline

ENV
The "ENV <variable_name> = <var_value> " solves the problem "ARG" has being that even after building the docker image.
typical use cage is setting variable used by application when they are starting like database directories or users.

"ENV"s downside is that it cant be overwritten in the build command. but it is possible to overwrite the variable in the run command. using "docker run --env <key>=<value> <image-name>"

Example
ENV Db_USER= pipeline_user

CMD psql -U $DB_USER
docker run --env postgres_user=test_db --env Postgres_password=test_deb postgres

Temporary containers
By default a container doenst get cleaned up even after stopping/exiting when you use the "docker run" command. when you just want a container to run a script this is problematic. to solve this you can use the "--rm" flag which removes the containers upon exit

Example
docker run --rm alpine:letets /bin/sh

Attaching files/directories to containers
For data to be available for multiple containers and for data to become presistent even after exiting. a bind-mount can be used. a bind-mount can be set to read-only or read/write access.
when a file or a directory is attached to a container they are not accessible to the host under the container is stopped.

the bindmount often used in the form of the "-v <source>:<destination>" flag you can use multiple "-v" flags per container

another option is the "--mount" flag which gives additional options

Example
docker run -v ~/html:/var/www/html nginx

Port mapping
Port mapping also called port fowarding/translation is the act of connecting an unique unused port on the host machine to a port on the container. this is done through using the "-p <host port>:<container port>" flag in the run command. there can be multiple "-p" flags in one run command if multiple ports need to be connected.

Alternatively "-P" upper case p can be used without any designated ports than a high port number is used to map the ports. (need to use docker ps -a to see which was used

Example
# choose the port yourselve
docker run -p 5501:80 nginx
# let docker choose the port
docker run -P pyserver

Docker file
To expose a port using a docker file the "EXPOSE <number>" instruction is used. optionally you can put "/tcp" or "/upd" behind the port number. you can use the "EXPOSE" instruction multiple times

exposing the port is generally used as a documentation method as it doesnt actually expose the ports withouth using the "-p" flag in the "docker run" command. (but without this users cant know which port should be mapped)

Example (docker file)
From python:3.11-slim
ENTRYPOINT ["python", "-mhttp.server"]
EXPOSE 8000

Presistent volumes
A volume is another way to store data in docker that is not related to container images or the host file system. volumes can be shared between multiple containers and i more powerful than the bind method but take more processing resources. volume mounts continue to exist till they are deleted themselves deleting the connected containers doesnt automatically delete the volume mount.

docker volume create
to create a volume mount the "docker volume create <volumename>" command is used.

Example
docker volume create sqldata

docker volume ls
to retrieve a list of all existing volume mounts the "docker volume ls" command is used. (it will retrieve just the names of the volumes)

Example
docker volume ls

docker volume inspect
to further retrieve metadata of a volume mount the "docker volume inspect <volume_name>" can be used.

Example
docker volume inspect sqldata

docker volume rm
to remove a volume the "docker volume rm" command is used.

Example
docker volume rm sqldata

attaching volumes to containers
to attach the volume to a container in the run command the "-v <volumename>:<destionationpath>:<options>" flag is used.
for options you can use e.g. "ro" to set it to read only

an alternative is using the "--mount" flag which has additional options.

Example
docker run -v sqldata:/data postgres

Docker networks
Docker can create networks to make it possible to be there communicate between containers, the host and other exernal systems.
Docker supports different networking types using drivers.
* bridge: default driver allows connections out, connection in if exposed.
* host: allows full communication between host and containers
* none: Isolate container from network communications

docker network ls & inspect
The "docker network ls" command is used to list all docker networks on the host. the "docker network inspect <networkname>" command can than be used to get further info about a network

Example
docker network ls
docker network inspect mynetwork

docker network create
the "docker network create <network name>" command is used to create a network

Example
docker network create mynetwork

docker network rm
The "docker network rm" command is used to remove a network

Example
docker network rm mynetwork

Attaching a container to a network
to attach a container to a net work using multiple methods.
* you can use the "--network <network name>" flag in a "docker run" command.
if the container is allready running you can use the "docker network connect <networkname> <container>" command

Example
docker run --network mynetwork ubuntu bash
docker network connect mynetwork ubuntu-B

Multi-stage builds

multi stage builds use multiple containers to build a single final container image. it typically has one or more build stage and final components are copied into this image.

the purpose of this is minimizing the size of the container image as the previous build steps are cleaned up afterwards automatically.

Muli stage builds use unique instructions in the docker file.including "AS <alias>" = is used to allow

 and "COPY --from=<alias>" = used to copy data from the previous stage container image into a following or final container image.

Example
#initial build stage
FROM ubuntu AS stage1
run apt install gcc -y
run make

#final stage
FROM alpine-base
COPY --from=stage1 /data_app /data_app
#run application on container start
CMD ["data_app"]

Multi-platform dockerfile
To ensure the dockerfile can be run on any plarform seperate tags and instructions can be used:

"--platform=$BUILDPLATFORM" flag is used in the "from" instruction to ensure the program is donwloaded for the platform the build is ran on.

to pass local environment variables into the docker build system "TARGETOS" and "TARGETARCH" are used which can be set using the "env" command.

to build a multi-platform build the "docker buildx --platform <linux/amd64, linx/arm64, ect> -t <appname> ." command needs to be used instead of "docker build"

prior to the build command you need to run "docker buildx create --bootstrap --use" command

Example (docker file)
FROM --platform=$BUILDPLATFORM golang:1.21 AS build

WORKDIR / src
COPY . .

ARG TARGETOS TARGETARCH

RUN env G00S=$TARGETOS GOARCH=$TARGETARCH go build -o /final/app .

FROM alpine
COPY --from=build /final/app /bin

Docker compose
Docker compose is an aditional command-line tool for docker used to define and manage multi-container applications. this is done by specifying containors networks and storage volumes in a single compose.yml/yaml file.

compose.yaml files have the big benifit of being easy to share and demo applications.

a typical yaml file is setup as follows

Example
#define the services (only required component)
services:
   #define the contaiers by name
   postgres:
        #container name (otherwise random if not defined)
        container_name: postgres
        #define the image to use and optionaly its version
        image: "postgres:latest"
         #optionally define the port forwarding
         ports:
             - "5432:5432"
         #what to do when a container crashes options (no, always, unless-stopped (most commun one), on-failure)
         restart: unless-stopped
         #condition defines how to decide when a resource is ready and fully started (service_started= resource has started normally is the default, service_completed_successfully = resource ran to completion such as initial configuration, service_healthy = resource meets a criteria defined by healthcheck can be defined in the docker compose file itself)
           condition: service_started
           # any resources like volums networks configs and secrets fall under the resources. they be assinged here or ocnce defined below in their sections can be referenced here e.g if multiple contianers use the same volume volume mount
           resource:.
               #volumes section be used to volume mount a host directory or file to a container directory
               volumes:
                    - <host directory>:<container directory>
               #another resource is networks
               networks:
                    network_name:
                        net1:
    #if there are more services defien them too
    webapp:
        image: "webapp"
        ports:
             - "8000:5000"
        #if one containers needs to be started before an other or needs to be stopped before anoter use (in reveres order)
        depends_on:
             -postgres
#optionally define any networks (to setup networks) multiple contaienrs might want to use and could reference from here
networks:
   ...
#optionally define volume sectionto control any volume mounting that  multiple contaienrs might want to use and could reference from here
volumes:
   ...
#optionally define configuration options without custom images that multiple contaienrs might want to use and could reference from here
configs:
   ...
#optionally the secret section provides options to handle passwords, tokes and api keys ect. that multiple contaienrs might want to use and could reference from here
secrets:
   ...

Starting a docker compose application
to start the compose.yaml applications you use the "docker compose up"  inorder to not block the entire terminal till all containers exit the "-d" flag can be used.

to use a yaml file located in a other directory than the working directory the "-f <yaml location>" flag can be used to to provide a file location.

generally the docker comose will create network by itself if not specified inside the .yaml file.

Example
docker compose up -d

status of running compose
you can check the status of a docker compose using "docker compose ls"

Example
docker compose ls

Docker compose stop /down
There are two ways to stop a running docker compose, each with their own benifites.

Docker compose stop
"docker compose stop" is the gentle way of stopping a docker compose. it stops the processes inside the containers but the container objects remain intact (thus will not update next time you up as they remain in the cache.

Example
docker compose stop

docker compose down
"docker compose down" is used to cleanup everything it stops all the containers and then deletes them, including the internal created network.the only thing that remains is the data if stored in volumes and the image files.
it is used to cleanup after a project is finished or not needed anymore.

Example
docke compose down

Docker compose logs
the "docker compose logs <optional resourcename> "  is used as a troubleshooting tool and gather output from all resources in the application. if you give it a resourcename it will only return the logs for that resource.

to see the live log of the containers the "-f" flag can be added to the command.

Example
docker compose logs

docker compose logs -f postgres

Docker compose top
"docker compose top" command is used for observation purposes as it will return the statuses,ids and uptimes of each of the running containers

Example
docker compose top
