Orchestration
Container orchestration comes in when you have multiple containers. orchesjtration is the automated management and coordination of multiple containers. for this an orchestrator is used. to achieve this a config file is written which describes the desired autocome. with this config file the orchestrator will allocate cpu memory ect over the containers that need it

To achieve this you can use docker swarm mode or kubernetes
Kubernetes often abbrivated as K8s. Kubernetes groups the containers that make up an application into logical units. Kubernetes operates as a distributed system, meaning that its various components are spread across different machines. These machines can be virtual or physical, hosted either in the cloud or on-premise.
While kubernetes can run docker images it does not use the docker engine to so but its own egine. docker is generally used for creating and updating docker images and starting the containers from such images

Kubernetes
Kubernetes consists out of serveral components

Pods
A pod is the smallest deployable unit in Kubernetes, consisting of one or more containers that share the same computing resources. Each pod serves as a unique environment in which applications run.

Nodes
One or more pods are grouped into a node, and each node serves as the host on which the pods run. Nodes are the smallest unit of computing hardware in Kubernetes, representing a pool of computing resources, whether virtual or physical, hosted in the cloud or on-premise (machines).

Control plane
The Control Plane manages nodes and pods and ensures efficient resource allocation by smartly assigning pods to available nodes. The Control Plane is also known as the brain of the cluster.

Kubernetes service
A cluster can have mulple services running on them. a service is a resource to expose network connectivity. its needed for when pods need to talk to each other or we want to talk to a pod a single service can connect to multiple pods even once seperated over multiple nodes.

Cluster
A Kubernetes cluster includes all previously mentioned components and is a group of up to multiple thousands of nodes. In a cluster, kuberentes keep track and organiazes which machine(Node) is running a particular application

Stateful applications
Stateful applications need pods that belogn together in a set they might work on different tasks and use different data. they save some internal state when interupted or stopped a new replica pod can read this saved state and continue operation from this state. An example is the hosting of a database that deliveres data to a frontend.

the kind tag in the manifest for a stateful application is "StatefulSet" what makes it different from the "Deployment" kind is that the pods are created in order and get predicatable names "pod-0, pod-1" ect making it possible for pods to take on different roles.

Manifests
 kuberenets objests are als ocalled kubernetes manifests. manifets are stored in the form of a .YAML file similar to docker compose yaml files. containing th objects and how they shoudl be configured, where they should be scheduled.

in here several types of tags can be found inlcuding:
apiVersion = defines the kubuernets API and version (for deploymnets this is apps/v1
kind = indicates it will create a deloyment
metadata = contains its owntags like name and labels
spec =  defines detailed specification of the deployment like
    replicas = number of replicated pods
    select =  used to defined how pods are managed
    template =  how new pods are created it uses this template to add labels, create containers with specific names, docker image and other ports (just like the docker yml)

Example
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80

Stateless application
A stateless application is an application is an application that does save an internal state or context of processed data, therefore its perfects for applications that just need to do a single query and return it. each pod of the application therefore has the exact same task.

the kind tag in the manifest is "Deployment"

Scaling kubernetes
scaling is a technique to add or remove resources. scale up increases the load and scale down saves resources. there are 2 ways to scale a pod you can change the number of replicas in the manifest or use the " scale deployment <deployment -name> --replicas <number>" command
similiarly for stateful application the command "kubectl scale statefulsets <deployment_name> --replicas (number)" is used. when scaling down the order of creation is usedso last created gets deleted first.

Example
#for stateless application
kubectl scale deployment web-server --replicas=5

# stateful application
kubectl scale statefulsets mysql-db --replicas=3

Kubernetes storage
The fundamental object for storage for uburnetes is Presistent volumens (PV) maintained in parallel to pods, their mapped to pods using presistent volume claims (pvc).

pvs allow data presistence when the pod is stopped, killed or estarted, they als enable the seperation of sotrage and computing.

There are two ways to setups presistent volumes.
1. manually by a kuberentes admin
2 dynamically by regular user.

Dynamic
Dynamic provisioning happens via storage classes (Sc) without human intervention. they define basic proporties of pvs to be dynamically provisioned. There are 3 objects that make storage work in kubernetes.

PresistentVolumeClaim = is what a pod uses to demand for presisted data
PresistentVolume = The pvc creates a presistentvolume for a pod claiming presistent volume
SotrageClass = what is used to define details like latency and back up strategy of the pv

in the manifest the declaration of a claim is done using the volumeMounts; tag. than under the volumes tag more details are given like the name and the presisitetnVolumeClaim itself

Example (pod with PresistentVolume)
apiVersion: v1
kind: Pod
...
spec:
  containers:
     ...
       volumeMounts:
        -name: pv-mydata
          mountPath: /mydata
   volumes:
   - name: pv-mydata
  presistentVolumeClaim:
     claimName: datacamp-pvc

Example (presistenVolumeClaim with StorageClass)
apiVersion: v1
kind: PersistentVolumeClaim
metadata;
   name: datacamp-pvc
spec:
   storageClassName: "standard"
   accessModes:
   - ReadWriteOnce
  resources:
      requests:
          storage: 5Gi

Load balancer
A load balancer is part of a kuberentes service, it main purpose is to distribute the load over pods to increase efficiency and lower reponse times as they are a type of service they can be specified in the spec section of the service manifest.

Example
apiVersion: v1
kind: Service
metadata:
    name: <service name>
spec:
    type: LoadBalancer
    selector:
           <key1>: <value1>
            <key2>: <value2>
